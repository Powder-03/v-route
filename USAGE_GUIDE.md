# v-route: User Integration Guide

Welcome to the **v-route** LLM Gateway. This guide provides all the necessary details on how to integrate and use the API to generate responses from large language models while taking advantage of our semantic caching layer.

## 🚀 Overview

The v-route gateway acts as a high-performance proxy between your applications and upstream LLMs (like OpenAI, Google Gemini, Anthropic, etc.). 

It automatically caches prompts that have a high semantic similarity (meaning they ask the same thing in a slightly different way). This dramatically reduces latency and saves API costs on repetitive queries.

---

## 🌐 Base URL

If you are running the gateway locally using Docker Compose, the API will be available on your local network via Nginx on port `80`.

**Base URL:** `http://localhost/`

*(Note: In a production environment, replace `localhost` with your actual domain or server IP address).*

---

## 🔌 API Endpoints

### 1. Generate Response
**Endpoint:** `POST /generate`

This is the primary endpoint for sending prompts to the LLM. 

#### Request Format
The endpoint expects a JSON payload containing a `user_id` and the `prompt`.

```json
{
  "user_id": "user-12345",
  "prompt": "What is the capital of France?"
}
```
* **`user_id`** (string, required): A unique identifier for the user making the request. Used for telemetry and analytics tracking.
* **`prompt`** (string, required): The actual question or instruction you want the LLM to answer.

#### Response Format
The endpoint returns a JSON payload containing the LLM's generated response and a flag indicating whether the response was served from the cache.

```json
{
  "response": "The capital of France is Paris.",
  "cache_hit": false
}
```
* **`response`** (string): The text generated by the LLM or retrieved from the cache.
* **`cache_hit`** (boolean): Will be `true` if the prompt was semantically similar to a previous prompt and served instantly from Redis, or `false` if the request had to be routed to the actual LLM.

## 💻 Example Code Snippets

### Using `cURL` (Terminal / Command Line)

```bash
curl -X POST http://localhost/generate \
     -H "Content-Type: application/json" \
     -d '{
           "user_id": "test_user_01",
           "prompt": "Explain quantum computing simply."
         }'
```

### Using Python (`requests` library)

```python
import requests

url = "http://localhost/generate"
payload = {
    "user_id": "dev_user_02",
    "prompt": "Write a python function to reverse a string."
}
headers = {
    "Content-Type": "application/json"
}

response = requests.post(url, json=payload, headers=headers)

if response.status_code == 200:
    data = response.json()
    print("LLM Output:", data["response"])
    print("Served from Cache?", data["cache_hit"])
else:
    print("Error:", response.text)
```

### Using JavaScript / Node.js (`fetch`)

```javascript
async function getLLMResponse() {
  const response = await fetch("http://localhost/generate", {
    method: "POST",
    headers: {
      "Content-Type": "application/json"
    },
    body: JSON.stringify({
      user_id: "web_client_03",
      prompt: "What are the benefits of semantic caching?"
    })
  });

  const data = await response.json();
  console.log("Response:", data.response);
  console.log("Cache Hit:", data.cache_hit);
}

getLLMResponse();
```

## 🧠 How Semantic Caching Works (What to Expect)

If you send the exact same prompt twice, the second query will return immediately with `"cache_hit": true`.

However, the gateway is "smart". If you send a structurally different but semantically identical prompt, it will *also* trigger a cache hit! 

**Example Workflow:**
1. **Query 1:** `"Can you explain how gravity works?"`
   * Response Time: ~2 seconds
   * `cache_hit`: `false` (Gateway reaches out to the actual LLM)

2. **Query 2:** `"How does the concept of gravity function?"`
   * Response Time: ~50 milliseconds
   * `cache_hit`: `true` (Gateway recognizes the semantic similarity >95% and returns Query 1's cached response).
